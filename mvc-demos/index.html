Commentary

Angular

  • Meticulous MVC pattern with focus on automation. Models auto-generate, views auto-bind, controllers are minimalist.
  • Bigger and more enterprise-feeling framework. Lots of docs to read through, making it slower to start with.
  • Seems to do a stupid amount of view processing. All view bindings update with every model change?
  • Self-contained; includes everything from DOM management to Ajax services and routing.

Backbone

  • Features an elegant model system with native REST services and built-in routing.
  • Small framework with clean and concise documentation. Fast to look over and start using.
  • Odd MV* pattern. Provides Model, Collection, and View, with amorphous relationship between components. Bad or good?.
  • Brute-force approach to views: regenerates dirty DOM from script templates.
  • Feels cluttered with dependencies.

Ember

  • Enterprise-level framework that tries to do everything, and turns into a mess.
  • Larger framework, or so it seems it with its long-winded and obtuse documentation. Slow to start with.
  • Attempts to solve a lot of the same problems as Angular and Knockout, but with less elegance.
  • BUTT-UGLY data binding. Renders script templates that will kick puppy dogs and make your mother cry.
  • Bulky framework who's features don't match its girth IMHO: 140kb + jQuery?

Knockout

  • Features a simple declarative data binding system for controlling complex views.
  • Framework is small and simple, yet packs some serious black magic features. Interactive tutorials provide a quick start.
  • Doesn't provide much beyond view controllers. Write your own data services.
  • Great for small, view-heavy apps with modest data requirements.
  • Light-weight, no dependencies, and fun to use. Definitely "knocks stuff out" quickly.

Spine

  • Very similar to Backbone. Great model system, native REST and routing.
  • Nice simplification of Models and Controllers from Backbone. Much more concise component relationships.
  • Documentation is long-winded in a mix of JS and CoffeeScript. Not as clean and easy to pick up as Backbone.
  • Takes the same brute-force approach for regenerating dirty views from template as Backbone.
  • Fragmented into components, making it potentially lighter, but more confusing to configure.
